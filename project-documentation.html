Section 1

Project Overview & Purpose

PLATFORM-PFE is a full-stack Cyber Threat Intelligence (CTI) SaaS platform built as a final-year engineering project (PFE). It is designed for a Managed Security Service Provider (MSSP) — a company that manages cybersecurity for multiple client businesses.

What Problem Does It Solve?

Security analysts spend hours manually searching threat feeds, correlating indicators, and writing reports for each client. This platform automates the entire process: collecting threat intelligence from external feeds, extracting indicators (IoCs), correlating them with client assets, and generating AI-powered threat reports — all without manual intervention.

Two Types of Users

🛡️ Admin MSSP (Security Analyst)

The MSSP employee who manages the entire platform. Sees all threats globally, manages all clients, configures data sources, validates IoCs, runs pipeline jobs, and generates reports for clients.

🏢 Client (End Customer)

A company that subscribes to the MSSP's service. Sees only the threats, alerts, IoCs, and reports relevant to their organization. Cannot see other clients' data.

Core Concept — Automated Threat Hunting Pipeline

The heart of the platform is a 7-step automated pipeline that runs continuously in the background:

#StepWhat Happens
1Initial ConfigLoad client profiles and pipeline settings
2Data CollectionFetch threat feeds from external CTI sources (MITRE, VirusTotal, etc.)
3NLP ExtractionAI extracts IoCs (IPs, domains, hashes) from raw threat text
4IoC ValidationValidate extracted indicators against reputation databases
5Internal CorrelationMatch IoCs against each client's assets (SIEM, EDR logs)
6Alert GenerationCreate alerts for clients whose assets match threat indicators
7AI Summary + ReportGenerate AI-written situation summary and remediation recommendations

The login page has multiple features:

  • Show/hide password toggle — eye icon button

  • Face ID / Biometric login — uses WebAuthn (browser fingerprint/face scan API)
  • Dev Quick Login — visible only in development mode, fills in demo credentials
  • Error handling — 401 = wrong credentials, 429 = rate limited, other = network error

Demo Accounts (DEV Mode Only)

client@demo.comclient123client/dashboard

Auth State — src/store/authStore.ts

Uses Zustand to store auth state globally. Key design decisions:

Access Token — In Memory Only

The short-lived JWT (15 min) is kept only in JavaScript memory. If the page refreshes, it is gone. This prevents XSS attacks from stealing it via localStorage.

Refresh Token — sessionStorage

The long-lived token (7 days) is stored in sessionStorage. This survives page refresh but is cleared when the browser tab closes — better security than localStorage.

Token Refresh Flow — src/services/api/apiClient.ts

The Axios interceptor handles token expiry automatically:

Request → attach Authorization header
    ↓
Response = 401 (token expired)?
    ↓ YES
Call /auth/refresh with refreshToken
    ↓
Got new accessToken?
    ↓ YES → retry original request with new token
    ↓ NO  → force logout → redirect to /login

Authentication Hook — src/hooks/useAuth.ts

This is what every page uses to interact with auth:

const { user, isAuthenticated, isAdmin, isClient, login, logout } = useAuth()

Routing & Access Control

How Navigation Works

React Router v7 handles all navigation. The app has three layers of protection before a user can see a page:

①
URL Match
React Router checks if the URL matches a defined route
②
ProtectedRoute Guard
Checks isAuthenticated — if false, redirect to /login
③
RoleGuard
Checks user role — if wrong role (e.g. client accessing /admin/*), redirect away
④
Page Renders
User has correct auth and correct role — the page component renders

Key Security Concept — Multi-Tenancy

🔒

Every client user has a clientId in their JWT token. When the backend receives a request from a client, it automatically filters all data by clientId. This means Client A can never see Client B's threats, IoCs, or alerts — even if they try to manually change the URL. The admin (no clientId) sees everything.

State Management & API Layer

Zustand Auth Store — src/store/authStore.ts

Stores who is currently logged in. Only one store in the whole app — auth state is the only truly global state needed.

State:
  user            → { id, email, name, role, clientId? }
  isAuthenticated → true | false
  accessToken     → string | null  (in memory only)
  refreshToken    → string | null  (persisted to sessionStorage)

Actions:
  setAuth(user, accessToken, refreshToken)  → call after login
  setAccessToken(token)                     → call after token refresh
  logout()                                  → clears everything

API Client — src/services/api/apiClient.ts

A pre-configured Axios instance that points to the FastAPI backend at http://localhost:8000/api.

Two interceptors are attached:

Request Interceptor

Before every request, reads accessToken from Zustand store and adds it as Authorization: Bearer <token> header automatically.

Response Interceptor

If a 401 error comes back (token expired), queues the failed request, calls /auth/refresh, gets a new token, then retries all queued requests.

Section 11

The 7-Step Automated Pipeline

This is the core innovation of the platform — a fully automated threat hunting pipeline that runs without human intervention. It is the central theme of the PFE project.

01
Initial Configuration
Load active client profiles, asset lists, and pipeline configuration settings. Determine which threat feeds to query for each client.
02
Data Collection
Fetch raw threat intelligence from external CTI sources: MITRE ATT&CK, VirusTotal, AlienVault OTX, Shodan, RSS feeds, and custom STIX/TAXII feeds. Raw data is saved to the database.
03
NLP Extraction
An AI/NLP model reads the raw threat text and extracts structured IoCs: IP addresses, domain names, file hashes (MD5/SHA256), URLs, and email addresses. Each extraction gets a confidence score (0-100%).
04
IoC Validation
Each extracted IoC is checked against reputation databases (VirusTotal, AbuseIPDB, etc.) to determine if it is a genuine threat indicator. Status is set to: validated / rejected / pending manual review.
05
Internal Correlation
Validated IoCs are matched against each client's internal security data (SIEM logs, EDR telemetry, firewall logs). If a client's network touched a known malicious IP, a correlation match is created.
06
Alert Generation
For each correlation match, an alert is created for the affected client. The alert includes: threat source, matched IoC, affected asset, severity level, and recommended immediate actions.
07
AI Summary + Report Generation
An AI model (Claude API / OpenAI) writes a natural-language situation summary for each client, prioritizes the top 3 actions, and compiles a full PDF report. The report is stored and made available for download in the client portal.

Pipeline in the Dashboard

The Admin Dashboard shows a live horizontal stepper visualizing the pipeline's current state:

  • Completed steps — green checkmark icon (CheckCircle2)
  • Active step — cyan pulsing glow + Loader2 spinning icon
  • Pending steps — gray, no icon animation

Recommendations — What to Add Next

Priority 1 — Critical
Build & Connect the FastAPI Backend
Everything is running on mock data. The entire platform is a frontend-only prototype. The next step is to scaffold the FastAPI project, create SQLAlchemy models for all 10 database tables, implement the auth endpoints (/auth/login, /auth/refresh, /auth/me), and connect the login page to a real JWT system. Once auth works, replace each page's mock data with real API calls one by one.

Tables needed: users, clients, threats, iocs, alerts, recommendations, reports, pipeline_jobs, nlp_extractions, correlations
Priority 2 — High
Implement Missing Admin Pages
ValidationPage: A review queue where analysts approve/reject IoCs with a reason. Uses a card layout with Approve (green) and Reject (red) buttons.
JobsPage: Real-time table of pipeline runs with status, duration, step progress, and error logs.
CorrelationPage: Rule builder UI — "IF IoC type is IP AND confidence > 80% THEN alert client X".
SourcesPage: CRUD interface for CTI feed URLs with enabled/disabled toggles and last-fetch timestamps.
Priority 2 — High
Implement All Detail Pages
ThreatDetailPage: Full threat profile with MITRE ATT&CK technique mapping, related IoCs list, affected clients, and activity timeline.
IocDetailPage: IoC enrichment view — VirusTotal score, WHOIS data, geographic location (for IPs), validation history with analyst notes.
AlertDetailPage: Alert investigation view — affected asset, matched IoC, MITRE technique, kill chain position, recommended actions with checkboxes.
Priority 3 — Medium
Add Toast Notifications
Currently there is zero feedback when a user performs an action (copy IoC, download report, validate an IoC). Add a toast library (e.g. react-hot-toast or sonner) and show success/error toasts for every user action. This dramatically improves usability and gives users confidence that their actions worked.
Priority 3 — Medium
Make the Topbar Search Functional
The search bar in the Topbar is a placeholder. Options: (1) make it a global search that queries the backend across threats, IoCs, and clients at once, or (2) make it context-sensitive — on the Threats page it filters threats, on the IoCs page it filters IoCs. Option 1 is more impressive for a PFE demo.
Priority 3 — Medium
Connect the 7-Step Pipeline to Real Backend Jobs
The "Launch Operation" button in the Topbar should call POST /api/jobs to trigger a real pipeline run. The pipeline stepper in the admin dashboard should poll GET /api/jobs/:id every 3 seconds and update the step statuses in real time. This is the most visually impressive demo feature for the PFE presentation.
Priority 4 — Lower
Mobile Responsiveness
The sidebar currently collapses to 64px on mobile but does not fully work as a mobile overlay. Add a hamburger menu that opens the sidebar as a full-screen overlay on small screens, and ensure all tables are horizontally scrollable on mobile.
Priority 4 — Lower
Export Features
Add CSV export to the Threats and IoCs tables (simple Blob download in the browser). Add PDF export for the AI Decision Support section. These are easy to implement and look great in a PFE presentation.
Priority 5 — Before Production
Security Hardening
Frontend-side rate limiting on the login form (disable for 30 seconds after 5 failed attempts). Auto-logout when the session expires and refresh fails. HTTPS-only warning when served over HTTP. Content Security Policy (CSP) headers on the backend.
